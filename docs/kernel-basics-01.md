## 1. 什么是操作系统内核

内核（kernel）是操作系统最核心的部分，直接运行在硬件之上。它负责：

- **管理硬件资源**：CPU、内存、磁盘、网卡等
- **提供抽象接口**：让应用程序不必直接操作硬件
- **隔离和保护**：确保一个程序的崩溃不影响其他程序

```
┌─────────────────────────────────────────────────────┐
│                   用户空间 (User Space)               │
│                                                       │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│   │  Shell   │  │ Firefox │  │  gcc    │  ...        │
│   └────┬────┘  └────┬────┘  └────┬────┘             │
│        │            │            │                    │
│════════╪════════════╪════════════╪════════════════════│
│        │     系统调用接口 (syscall)  │                │
│        ▼            ▼            ▼                    │
│                   内核空间 (Kernel Space)              │
│                                                       │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│   │ 进程管理  │ │ 内存管理  │ │   VFS    │  ...       │
│   └──────────┘ └──────────┘ └──────────┘            │
│                                                       │
│════════════════════════════════════════════════════════│
│                   硬件 (Hardware)                      │
│   CPU    RAM    Disk    NIC    GPU    ...             │
└─────────────────────────────────────────────────────┘
```

**用户空间 vs 内核空间**：

| 特性 | 用户空间 | 内核空间 |
|------|---------|---------|
| 权限 | 受限（Ring 3） | 完全（Ring 0） |
| 内存访问 | 只能访问自己的虚拟地址空间 | 可以访问所有物理内存 |
| 硬件访问 | 不可直接访问 | 可以直接操作硬件 |
| 崩溃影响 | 进程被杀死 | 整个系统崩溃（kernel panic） |
| 切换方式 | 通过系统调用进入内核 | 通过返回指令回到用户态 |

> **Ring 0 和 Ring 3** 是 x86 CPU 的特权级别。Ring 0 拥有最高权限，可以执行任何指令；Ring 3 是最低权限，不能直接操作 I/O 端口或修改页表。内核运行在 Ring 0，应用程序运行在 Ring 3。

## 2. 宏内核 vs 微内核

操作系统内核的架构主要分两种流派：

### 宏内核（Monolithic Kernel）

所有核心功能（进程调度、内存管理、文件系统、网络协议栈、设备驱动）都运行在**同一个地址空间**，共享同一个 Ring 0 特权级别。

```
┌──────────────────────────────────────────┐
│              内核空间 (Ring 0)             │
│                                           │
│  进程管理 ←→ 内存管理 ←→ VFS ←→ 网络     │
│       ↕          ↕         ↕      ↕       │
│     调度器     页表管理   ext4   TCP/IP    │
│       ↕          ↕         ↕      ↕       │
│              设备驱动层                    │
│  键盘驱动  磁盘驱动  网卡驱动  GPU 驱动   │
│                                           │
│  所有组件共享地址空间，函数调用即可通信     │
└──────────────────────────────────────────┘
```

**优点**：组件之间直接函数调用，性能极高
**缺点**：一个驱动的 bug 可以搞崩整个内核

**Linux 就是宏内核。** 但它通过**可加载内核模块（Loadable Kernel Module, LKM）**实现了一定的灵活性——驱动可以动态加载/卸载，不需要重新编译整个内核。

### 微内核（Microkernel）

只把最基本的功能（进程间通信 IPC、基本调度、内存地址空间管理）放在内核空间，其余功能（文件系统、网络、驱动）都在用户空间运行。

```
┌──────────────────────────────────────────┐
│              用户空间 (Ring 3)             │
│                                           │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │文件   │ │网络   │ │磁盘   │ │显示   │   │
│  │服务   │ │服务   │ │驱动   │ │驱动   │   │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘   │
│     │  IPC   │  IPC   │  IPC   │         │
│═════╪════════╪════════╪════════╪═════════│
│              内核空间 (Ring 0)             │
│     ┌──────────────────────────┐         │
│     │  IPC + 调度 + 地址空间    │         │
│     └──────────────────────────┘         │
└──────────────────────────────────────────┘
```

**优点**：组件隔离好，一个驱动崩溃可以重启而不影响系统
**缺点**：组件通信走 IPC，性能开销大

代表：MINIX、QNX、seL4。

### 对比总结

| 特性 | 宏内核 (Linux) | 微内核 (MINIX) |
|------|---------------|----------------|
| 内核大小 | 大（数百万行代码） | 小（几万行） |
| 性能 | 高（直接函数调用） | 低（IPC 开销） |
| 可靠性 | 驱动 bug 可崩溃内核 | 服务可独立重启 |
| 代表系统 | Linux, FreeBSD, Windows NT | MINIX, QNX, seL4 |
| 可加载模块 | 支持（LKM） | 天然组件化 |

> **Rust-for-Linux 的意义**：Linux 作为宏内核，所有代码共享地址空间，一个驱动的内存错误就可能导致 kernel panic。Rust 的内存安全特性可以在**编译期**消除大量此类 bug，这对宏内核架构特别有价值。

## 3. Linux 的五大子系统

Linux 内核可以分为五大核心子系统：

```
                    系统调用接口
                         │
         ┌───────┬───────┼───────┬───────┐
         ▼       ▼       ▼       ▼       ▼
    ┌────────┐┌────────┐┌─────┐┌─────┐┌──────────┐
    │ 进程   ││ 内存   ││     ││     ││ 设备     │
    │ 管理   ││ 管理   ││ VFS ││ 网络 ││ 驱动     │
    │        ││        ││     ││     ││          │
    │ 调度器 ││ 页表   ││ext4 ││TCP  ││ 字符/块  │
    │ fork() ││ slab   ││tmpfs││UDP  ││ 网络设备 │
    │ clone()││ mmap   ││proc ││IP   ││ USB/PCI  │
    └────────┘└────────┘└─────┘└─────┘└──────────┘
         │       │        │      │        │
         └───────┴────────┴──────┴────────┘
                         │
                    硬件抽象层
```

### 3.1 进程管理（Process Management）

负责程序的执行和调度。

- **进程** = 运行中的程序 + 它的资源（内存、打开的文件、信号处理器等）
- **task_struct** 是内核中表示进程/线程的核心数据结构，包含 PID、状态、调度信息、内存映射等
- **调度器**（CFS, Completely Fair Scheduler）决定哪个进程获得 CPU 时间
- 关键系统调用：`fork()`（创建进程）、`exec()`（加载程序）、`wait()`（等待子进程）、`exit()`（退出）

```c
// task_struct 简化示意（实际有数百个字段）
struct task_struct {
    pid_t                 pid;          // 进程 ID
    volatile long         state;        // 运行状态
    struct mm_struct      *mm;          // 内存映射
    struct files_struct   *files;       // 打开的文件
    const struct cred     *cred;        // 权限凭证
    struct list_head      tasks;        // 全局进程链表
    // ... 还有很多
};
```

### 3.2 内存管理（Memory Management）

负责物理内存和虚拟内存的管理。

- **虚拟内存**：每个进程都有独立的虚拟地址空间（32 位系统 4GB，64 位系统 128TB）
- **页表**：将虚拟地址映射到物理地址，由 MMU 硬件实现
- **slab 分配器**：针对内核中频繁分配的小对象（如 task_struct）做优化
- **伙伴系统**（Buddy System）：管理物理页面的分配和释放
- 关键系统调用：`mmap()`（内存映射）、`brk()`（调整堆大小）、`mprotect()`（设置内存权限）

### 3.3 虚拟文件系统（VFS, Virtual Filesystem）

VFS 是一个抽象层，为上层提供统一的文件操作接口，下层对接不同的文件系统。

```
应用程序:    open()    read()    write()    close()
               │         │         │          │
VFS 层:    ┌───┴─────────┴─────────┴──────────┴───┐
           │     struct file_operations            │
           │     .open  .read  .write  .release    │
           └───┬─────────┬─────────┬───────────────┘
               │         │         │
具体实现:    ext4      tmpfs     procfs    NFS ...
```

**"一切皆文件"** 是 Unix/Linux 的核心哲学：
- 普通文件 → ext4、XFS 等文件系统
- 设备 → `/dev/sda`、`/dev/tty` 等设备节点
- 进程信息 → `/proc/` 虚拟文件系统
- 内核参数 → `/sys/` sysfs 文件系统
- 管道、socket 也通过文件描述符访问

### 3.4 网络子系统（Networking）

实现完整的网络协议栈。

- **Socket 层**：应用程序的网络接口（`socket()`、`bind()`、`listen()`、`accept()`）
- **传输层**：TCP、UDP
- **网络层**：IP、路由
- **链路层**：以太网帧、ARP
- **Netfilter**：防火墙框架（iptables/nftables 的底层）

### 3.5 设备驱动（Device Drivers）

设备驱动占内核代码量的 **~70%**，是最大的子系统。

三种类型：
- **字符设备**：按字节流读写（键盘、串口、终端）
- **块设备**：按块随机读写（硬盘、SSD）
- **网络设备**：通过网络协议栈收发数据（网卡）

```
drivers/
├── block/       # 块设备驱动
├── char/        # 字符设备驱动
├── gpu/         # GPU 驱动（nova 用 Rust 写）
├── net/         # 网络驱动
│   └── phy/     # PHY 驱动（有 Rust 实现）
├── usb/         # USB 驱动
├── pci/         # PCI 总线驱动
└── ...          # 更多
```

> **Rust-for-Linux 目前的主战场就是驱动**。由于驱动代码量大、bug 密度高，用 Rust 重写或新写驱动是 RFL 最有价值的方向。目前已有的 Rust 驱动包括 PHY 网络驱动和 nova GPU 驱动。

## 4. 系统调用：用户态与内核态的桥梁

应用程序无法直接调用内核函数。它们通过**系统调用（syscall）** 请求内核服务。

### 系统调用流程（以 x86_64 为例）

```
用户空间                                内核空间
┌──────────────┐                  ┌──────────────────┐
│  printf("hi") │                  │                  │
│      │        │                  │                  │
│      ▼        │                  │                  │
│  write(1,     │   syscall 指令   │  entry_SYSCALL   │
│    "hi", 2)   │ ───────────────► │  _64             │
│               │                  │      │           │
│  用户态代码    │  ① RAX=1(write) │      ▼           │
│  被暂停       │  ② RDI=1(fd)    │  sys_call_table  │
│               │  ③ RSI=buf      │  [__NR_write]    │
│               │  ④ RDX=2(len)   │      │           │
│               │                  │      ▼           │
│               │   sysret 指令    │  ksys_write()    │
│  恢复执行     │ ◄─────────────── │      │           │
│  返回值=2     │  RAX=2(写入字节) │  vfs_write()     │
└──────────────┘                  └──────────────────┘
```

**关键步骤**：

1. **用户程序调用 C 库函数**：`write(fd, buf, len)`
2. **C 库设置寄存器**：系统调用号放 RAX，参数放 RDI/RSI/RDX/R10/R8/R9
3. **执行 `syscall` 指令**：CPU 切换到 Ring 0，跳转到内核入口点
4. **内核查表执行**：通过 `sys_call_table[RAX]` 找到对应的内核函数
5. **执行完毕返回**：通过 `sysret` 指令返回用户态，结果放在 RAX

### 常见系统调用

| 系统调用 | 编号 | 所属子系统 | 功能 |
|---------|------|-----------|------|
| `read` | 0 | VFS | 从文件描述符读取 |
| `write` | 1 | VFS | 写入文件描述符 |
| `open` | 2 | VFS | 打开文件 |
| `close` | 3 | VFS | 关闭文件描述符 |
| `fork` | 57 | 进程管理 | 创建子进程 |
| `execve` | 59 | 进程管理 | 执行程序 |
| `mmap` | 9 | 内存管理 | 内存映射 |
| `socket` | 41 | 网络 | 创建套接字 |
| `ioctl` | 16 | 设备驱动 | 设备控制 |

> Linux x86_64 目前有约 **450+** 个系统调用。完整列表见内核源码 `arch/x86/entry/syscalls/syscall_64.tbl`。

## 5. 观察内核：/proc 与 strace

### 5.1 /proc 文件系统

`/proc` 是一个**虚拟文件系统**（pseudo-filesystem），不存在于磁盘上。它是内核向用户空间暴露信息的窗口。

```bash
# 查看 CPU 信息
cat /proc/cpuinfo

# 查看内存使用
cat /proc/meminfo

# 查看内核版本
cat /proc/version

# 查看某个进程的信息（PID=1 是 systemd/init）
ls /proc/1/
# cmdline    当前命令行
# maps       内存映射
# fd/        打开的文件描述符
# status     进程状态摘要
# stack      内核栈（需要 root）

# 查看系统负载
cat /proc/loadavg
```

`/proc` 的每个"文件"都是内核在你读取时**动态生成**的内容。它不占磁盘空间，是了解内核运行状态的最直接方式。

### 5.2 strace：追踪系统调用

`strace` 可以追踪一个程序执行的所有系统调用，帮助理解程序如何与内核交互。

```bash
# 追踪 ls 命令的系统调用
strace ls /tmp

# 输出示例（简化）：
# execve("/usr/bin/ls", ["ls", "/tmp"], ...) = 0
# openat(AT_FDCWD, "/tmp", O_RDONLY|O_DIRECTORY) = 3
# getdents64(3, ..., 32768)     = 120
# write(1, "file1.txt\nfile2.txt\n", 20) = 20
# close(3)                       = 0
# exit_group(0)                  = ?
```

**解读上面的输出**：
1. `execve`：加载并执行 `/usr/bin/ls`
2. `openat`：打开 `/tmp` 目录，返回文件描述符 3
3. `getdents64`：读取目录内容
4. `write`：将结果写到标准输出（fd=1）
5. `close`：关闭目录
6. `exit_group`：进程退出

```bash
# 只看特定类型的系统调用
strace -e trace=open,read,write ls /tmp

# 统计各系统调用的次数和耗时
strace -c ls /tmp

# 追踪正在运行的进程（需要 PID）
strace -p 12345
```

> **strace 的原理**：它使用 `ptrace` 系统调用附着到目标进程上，每当目标进程执行系统调用时，`ptrace` 会暂停目标进程并通知 strace，strace 记录调用信息后再让目标进程继续执行。

### 练习 1：绘制子系统交互图

画一张图，展示当你运行 `curl https://example.com > output.html` 时，以下子系统如何协作：

- 进程管理（fork + exec）
- 网络子系统（socket、TCP/IP）
- VFS（写文件）
- 内存管理（分配缓冲区）

### 练习 2：探索 /proc

在你的 Linux 开发机上执行以下操作：

```bash
# 1. 查看你的 shell 进程的内存映射
cat /proc/$$/maps

# 2. 查看系统当前有多少进程
ls -d /proc/[0-9]* | wc -l

# 3. 查看内核命令行参数
cat /proc/cmdline

# 4. 查看已加载的内核模块
cat /proc/modules | head -20
```

回答：`/proc/$$` 中的 `$$` 代表什么？

### 练习 3：用 strace 追踪系统调用

```bash
# 1. 追踪 echo "hello" 并识别它使用了哪些系统调用

# 2. 统计 ls / 的系统调用
strace -c ls /

# 3. 找出 cat /etc/hostname 使用的第一个和最后一个系统调用
strace cat /etc/hostname 2>&1
```
